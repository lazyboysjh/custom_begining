# 圣堂初遇：全题材随机开局与 150 人角色库设计

## 目标

将现有“选择相遇方式与初始角色”的封面流程改造为以未知揭晓为核心的随机开局系统，并把女性角色库从 78 人扩充至 150 人。

本次设计遵循以下确定要求：

- 圣堂的净化体系、跨界节点、仪式规则与核心据点身份保持不变。
- 圣堂的建筑、组织名称和社会身份可以随时代与题材改变。
- `user` 的能力是唯一由玩家决定的开局核心信息；支持预设、手动输入和单独随机。
- 初遇角色、时代、故事类型、故事氛围、圣堂形态和世界融入方式均可随机。
- 女主不受原作时代或题材限制，所有 30 种故事类型均可参与随机。
- 封面不允许玩家预先选择或查看本局初始女主，女主只在生成过程中揭晓。
- 本阶段不新增、校验、改写或筛选年龄规则；相关统一规范属于后续独立任务。

## 产品流程

### 普通开始

1. 玩家进入封面。
2. 玩家设置 `user` 能力：选择预设、手动输入或仅随机能力。
3. 玩家点击“生成开局”。
4. 系统一次性抽取女主、时代、故事类型、氛围、圣堂形态和世界融入方式。
5. 生成页播放女主揭晓动画，同时保留真实的文字生成进度。
6. 模型根据完整随机配置生成首段剧情与 MVU 更新。
7. 事务提交成功后替换封面楼层并进入剧情。

### 一键随机开始

“一键随机开始”不要求填写任何表单：它先随机 `user` 能力，再执行与普通开始相同的抽取、揭晓、生成和提交事务。

随机结果一经进入生成事务即锁定。取消或失败重试时可以重新抽取；成功生成前不提供女主重抽按钮，保持未知开局的完整性。

## 随机模型

随机配置只保留六个真正改变故事结构的维度：

```text
user能力（玩家决定或一键随机）
+ 初遇女主
+ 时代与世界
+ 故事类型
+ 故事氛围
+ 圣堂形态
+ 世界融入方式
```

污秽类型、症状、秘密、开场地点、具体冲突和场景细节不做独立表单字段，由模型根据完整组合推导，避免把随机系统拆成大量低价值参数。

所有女主对 30 种故事类型等概率开放，不根据原作题材排除候选。组合后使用兼容规则生成合理解释，而不是回退到原作时代。

## 动态圣堂

圣堂固定保留四项本质：

1. `user` 所属或能够调用的净化体系。
2. 连接不同时代、世界与精神空间的异常节点。
3. 圣物、仪式、契约和污染所遵循的规则来源。
4. 能够成为长期据点、任务来源或最终归处的核心存在。

圣堂外形根据世界生成，例如国师府、道观、镇魔司、荒山古刹、租界教堂、夜间诊所、异常事务所、地下避难所、移动医疗站、意识净化中心、星舰礼拜舱、魔法神殿、问心殿、冥府渡口或梦境圣堂。

动态形态不能只改名称。提示词必须同步改变圣堂的社会职能、可用资源、仪式表现、外界态度和空间结构。

## 三十种故事类型

1. 宫廷权谋：继承、派系和秘密交易。
2. 江湖恩怨：门派、旧仇和追杀。
3. 宗门修行：试炼、突破和宗门竞争。
4. 历史风云：时代事件与个人命运交汇。
5. 战争谍报：潜伏、策反和情报争夺。
6. 都市悬疑：失踪、跟踪和连续异常事件。
7. 刑侦探案：证据、嫌疑人与限时破案。
8. 犯罪卧底：双重身份和阵营选择。
9. 法庭审判：通过调查与辩论改变判决。
10. 医疗危机：未知病症、隔离和伦理抉择。
11. 校园青春：社团、比赛、流言和成长。
12. 娱乐圈风波：剧组、舆论、资本和身份包装。
13. 职场博弈：晋升、项目和组织斗争。
14. 豪门商战：继承权、并购和家族秘密。
15. 竞技赛事：擂台、联赛、选秀或生存竞赛。
16. 经营建设：共同经营圣堂、领地、组织或店铺。
17. 夺宝冒险：遗迹、机关和多方争夺。
18. 公路旅程：护送、追踪和沿途事件。
19. 荒野求生：资源匮乏与自然威胁。
20. 灾难逃亡：灾害、瘟疫或城市崩坏。
21. 末日生存：文明崩溃后的据点与派系。
22. 密闭惊魂：列车、孤岛、飞船或封锁建筑。
23. 黑暗童话：诅咒规则、象征世界和残酷代价。
24. 神话志怪：神明、妖怪、因果和民间传说。
25. 奇幻史诗：王国战争、魔法和世界危机。
26. 克苏鲁调查：不可理解的存在与认知侵蚀。
27. 无限副本：规则关卡、任务目标和淘汰机制。
28. 赛博犯罪：意识入侵、记忆交易和企业统治。
29. 星际冒险：星舰、殖民地、外星文明和宇宙遗迹。
30. 时空悖论：时间循环、平行世界和历史改写。

每种故事类型必须提供不同的核心目标、失败条件和推进方式。故事氛围不能替代故事结构。

## 故事氛围

每局只抽取一个主氛围，不叠加多个近义标签：

- 禁忌拉扯
- 敌对转合作
- 契约关系
- 强制共处
- 慢热救赎
- 悬疑博弈
- 危险共生
- 黑暗童话
- 身份反差
- 日常暧昧

氛围只约束关系节奏、叙事张力和表达风格，不预设女主服从、爱慕或失去独立目标。

## 世界融入方式

系统内部抽取一种世界融入逻辑，不在封面增加玩家选项：

- 原生重构：女主本来就生活在当前世界，拥有符合时代的新身份。
- 时空迁移：保留原作记忆，通过事故、召唤或圣堂节点来到这里。
- 转生再临：拥有新的生平，同时保留部分人格、能力或记忆碎片。
- 身份覆盖：进入当前世界中另一个“自己”的身份。
- 世界融合：原作世界与当前时代发生局部重叠。
- 圣堂召回：她过去与圣堂存在契约或未完成事件。
- 历史改写：世界认为她一直存在，只有少数人察觉异常。
- 任务投放：她主动跨界执行追捕、调查、逃亡或夺取任务。

生成提示词必须自然回答：

1. 女主为何出现在当前世界。
2. 她在当前时代具有怎样的身份与社会关系。
3. 她为何与当前形态的圣堂发生联系。
4. 她此刻主动追求什么。
5. 污秽或异常如何进入这条因果链。

解释必须通过事件、对话和环境进入正文，不能写成设定说明书。不得把“突然穿越”或“失忆醒来”作为默认答案。

## 角色跨时代适配

跨时代改写采用“核心不变、表现重构”：

- 保留原作核心性格、语言习惯、价值冲突、独立目标和行为能力。
- 保留角色最具辨识度的能力功能；表现形式可以适配世界。
- 保留原作中的重要经历如何塑造她，但允许经历在新世界拥有功能相近的对应事件。
- 不允许只保留姓名和外貌，把角色写成通用模板。
- 不允许因 `user` 的牧师、净化或主角身份而默认信任、爱慕或服从。

例如电磁控制在古代可以重构为雷法，在赛博世界可以重构为电网与义体控制，但其战斗逻辑和角色对能力的态度应保持可辨认。

## 角色库扩充

### 范围

角色总数从 78 人扩充至 150 人。范围包含用户指定的动漫、游戏、网络小说、古典文学女性角色及现实公众人物。

本阶段不处理统一成年规则。现实公众人物只整理可核实的公开经历、作品形象、职业特征与公开表达风格，不推断或编造私人生活、私密关系和未公开心理。

### 资料来源优先级

每名新增角色建立独立来源记录，按以下顺序研究：

1. 原著小说、漫画、动画、游戏文本、影视正片及官方角色资料。
2. 原作者访谈、官方设定集、出版社、制作方或经纪团队公开资料。
3. 能够交叉验证的百科、作品资料站和高共识角色百科。
4. 原作没有覆盖的表达细节，才参考具有广泛共识的网络同人理解。

同人资料只能补足语气、日常反应或跨题材灵感，不能覆盖原著明确事实。若资料互相冲突，以原著与官方资料为准；无法确认的内容写成克制推断，不作为既定事实。

来源记录保存于独立研究文件，不把 URL 和研究说明注入最终角色提示词。

### 单角色必填质量

每名角色继续使用现有结构，并满足以下要求：

- `id`：稳定、唯一、ASCII 标识。
- `name`：通行简体中文名。
- `aliases`：常用译名、称号和必要外文名，不能混入其他角色。
- `work`、`year`、`age_note`：按资料记录，本阶段不作统一年龄处理。
- `appearance`：至少包含发型、眼神或表情、服饰、体态和标志物，避免只写颜色。
- `blurb`：一句能够区分角色的定位，不使用“温柔美丽、善解人意”等空泛组合。
- `intro`：说明身份、关键经历、性格矛盾、能力或专长、行为方式和可辨识细节。
- `work_intro`：简要说明原作世界、角色位置及与主线的关系。
- `background`：至少三条，分别覆盖经历、核心矛盾和能力或社会关系。
- `filth_seed`：必须由角色经历或能力自然延伸，不能批量复用黑纹、低语等模板。
- `relations`：至少三条，包含与 `user` 初见基线、社交反应和明确的人格边界。
- `accent`：用于封面识别的稳定主题色，保持文本对比度。

新增角色还必须能够回答四个审查问题：她主动想要什么、她拒绝什么、她如何说话、她在没有 `user` 时会做什么。

### 批次与质量门

新增 72 人分为四批，每批完成后独立审查，再进入下一批：

1. 动漫与游戏角色。
2. 中国网络小说角色。
3. 古典文学、神话和历史文学形象。
4. 现实公众人物公开形象。

每批必须通过：数量与唯一性、别名串角色、作品归属、必填字段、重复文案、人物自主性、角色语气、污秽种子差异、世界书条目同步和构建注入检查。

## 生成页与揭晓动画

生成页延续 C2 玻璃材质，并增加“圣堂启示”揭晓区：

1. 圣堂标记或当前时代对应印记被点亮。
2. 候选角色名与作品信息快速掠过，但不完整暴露整个角色库。
3. 速度逐渐降低，最终角色定格并显示姓名、作品和一句角色定位。
4. 下方保留真实文字进度、当前生成状态和取消按钮。

揭晓动画与模型生成并行，但状态相互独立。动画结束不能伪装成生成完成；生成完成也要保证最终角色至少清晰展示一个短暂定格阶段。

手机端减少同时出现的候选文字数量，避免拥挤；`prefers-reduced-motion` 下使用淡入和定格替代高速滚动。

## 状态栏

状态栏不增加大面积新面板，只补充与随机开局直接相关的信息：

- 顶部场景信息增加故事类型、故事氛围和当前圣堂形态。
- 多角色切换继续以 `世界.出场角色` 数组为唯一在场来源。
- 当前角色面板保持现有数值、心态、目标和关系信息。
- `user` 能力以紧凑摘要显示，不展示生成提示词或内部随机字段。
- 世界融入方式、隐藏动机、真实污秽来源等仅供模型使用，不能提前显示给玩家。

在桌面端利用横向空间并排展示场景摘要和角色切换；手机端使用单行可滚动角色栏，故事类型与氛围允许换行但不挤压操作按钮。

## MVU 与数据流

`世界` 增加可持久化的公开开局信息：

- `时代背景`
- `故事类型`
- `故事氛围`
- `圣堂形态`
- `开局摘要`
- `出场角色`

`主角.能力摘要` 保存玩家最终选定或一键随机得到的能力。世界融入方式作为生成时配置进入提示词，除非剧情已经公开，否则不进入状态栏。

开局仍采用现有事务边界：抽取配置后锁定 `generation_id`，生成期间不提前写入 MVU；生成和解析全部成功后，以单次事务写入角色 record、公开世界字段和 `世界.出场角色`。必须先创建 `角色.${女主名}`，再把姓名加入出场数组。

后续同场景新增角色时继续向 `世界.出场角色` 追加，不能覆盖已有角色；每个角色数值和目标独立更新。

## 测试与发布

- 数据测试：150 人数量、ID/姓名唯一、必填字段、别名交叉污染、来源记录覆盖率。
- 内容测试：原著事实抽查、重复句检测、空泛模板检测、自主性与边界规则。
- 随机测试：30 种故事类型均可抽中；角色不因原作题材被排除；一键随机能够生成完整配置。
- 组合测试：固定随机种子批量生成组合，检查时代、圣堂形态、角色身份与故事目标具备完整因果链。
- MVU 测试：新增字段可迁移、增量解析幂等、先建角色再追加出场数组、多角色独立更新。
- UI 测试：封面不暴露女主选择；揭晓动画、取消、失败、重试与 reduced-motion 状态完整。
- 浏览器测试：360px、768px、1280px 下无横向溢出、按钮裁切、文本遮挡和控制台错误。
- 整卡审查：源码、世界书、角色库、卡内嵌入数据、固定 CDN 和生成事务全部一致。
- 发布顺序：静态资源提交并推送，卡内 CDN 固定该 commit，重建角色卡并再次提交推送，最后验证固定 CDN 返回 200。

## 非目标

- 本阶段不实现统一成年规则。
- 本阶段不增加年龄筛选、年龄重写或按年龄限制随机组合。
- 不把来源链接、构建信息、随机种子和隐藏生成配置显示给玩家。
- 不为每种污秽症状、场景秘密或关系细节增加独立表单控件。
